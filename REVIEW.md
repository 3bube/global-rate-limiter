# Qualification Review — Global Rate Limiter as a Service

Reviewer: senior engineering review, hiring-bar standard.
Scope: the submission as received (baseline commit `Baseline: candidate submission as received`); all fixes described below were applied on top of that baseline in this repo.

## Requirement-by-requirement verdict (as submitted, before fixes)

| # | Requirement | Status | Penalty rationale |
|---|-------------|--------|-------------------|
| 1 | Per-client limits | **Partially met** | Limits work and are per-client, but they were baked into the Docker image (`COPY clients.json`) — changing Client A from 100 to 200 req/min required an image rebuild and redeploy. Worse, an *unknown* clientId silently received a default 60/min bucket, so any caller could mint unbounded per-key state in Redis with invented IDs. The file was also never validated (a negative limit would load silently). Would raise all three in the debrief; the default-bucket behavior borders on failing. |
| 2 | Cluster-accurate checks | **Met** | The atomic Lua script + hash-tagged key (`bucket:{clientId}`) is the right design, NOSCRIPT reload is handled, and the race test (50 concurrent callers on 50 connections against limit=10, asserting exactly 10 admits) genuinely proves the atomicity property — Redis serializes scripts regardless of how many processes the callers live in, so the single-Node-process caveat is acceptable. Minor: `X-RateLimit-Reset` is epoch **ms** where the ecosystem convention is seconds — worth a comment only. |
| 3 | Few-ms checks | **Met** | One EVALSHA round trip, in-memory config lookup, hard 75 ms timeout so a hanging Redis can't stall the caller, and a k6 test asserting p95 < 50 ms end-to-end. Noted, not penalized: on a timeout the in-flight EVALSHA may still consume tokens while the request is also admitted via fail-open — bounded over-counting during degradation, a defensible trade. |
| 4 | Fail-safe when Redis is down | **Partially met** | The circuit breaker + fail-open core is correct and tested against both refused connections and hangs (the timeout wrapper covers the "Redis hangs" case most submissions miss). But the fail-open response returned `limit: -1, remaining: -1` — a real consumer parsing `X-RateLimit-Remaining` as an unsigned integer breaks the moment Redis blips, which is exactly when you least want client-side surprises. The in-memory registry already knew the real limit; not using it was a contract bug, would raise in the debrief. Also: `server.ts` had **no** graceful shutdown (SIGTERM = dropped in-flight requests, leaked connections) while `analyticsWorker.ts` handled signals — inconsistent lifecycle handling. Minor: HALF_OPEN admits unlimited concurrent trial calls (thundering herd on recovery). |
| 5 | Analytics logging off the hot path | **Partially met** — two findings here are **incident-class** | The architecture (fire-and-forget XADD → consumer group → batched inserts) is the right shape and genuinely keeps Postgres off the hot path. But: **(a)** the stream had no MAXLEN — if the worker dies or falls behind, `usage-events` grows until Redis OOMs, taking down the *rate limiter itself*; silent, slow, and fatal. **(b)** a worker crashing after XREADGROUP but before XACK strands those events in the pending list *forever* — no XAUTOCLAIM/XCLAIM anywhere; that is silent billing-data loss. **(c)** a crash after INSERT but before XACK causes redelivery and **duplicate billing rows** — no idempotency. For a service whose stated purpose includes billing, (a)+(b)+(c) together would fail this requirement outright in a production review. |
| 6 | Dashboard with complex filters | **Partially met** | 10/15/30-day trend graphs exist and work. But "complex filters" was two dropdowns (client, day-window) — minimally satisfied at best. "Avg response time" is actually the *limiter's own check latency* (sub-ms), not anything a client would recognize as their response time; the label didn't say so. And the headline stat computed an **unweighted mean of per-day averages** — a nearly idle day skews the number; that's a correctness bug in a billing-adjacent metric. Also: Chart.js loads from a CDN (dashboard dies without internet), and every dashboard API endpoint was world-readable. |
| — | **Authentication / authorization** | **Not met** | Nothing on `/v1/check`, `/v1/clients`, or `/v1/clients/:id/usage`. Anyone who can reach the service can **spend any client's quota** (every check consumes a token — this is a griefing vector against paid third-party API budgets, i.e. directly against the money this service exists to protect) or read any client's usage. An unauthenticated endpoint in front of paid APIs **fails on its own** in a hiring-bar review. |
| — | Self-protection of the limiter's API | **Not met** | The service that exists to prevent abuse had no protection against its own abuse. Would raise in the debrief. |
| — | Required output: ZIP contents | **Met, with dings** | README is genuinely good (run steps, test steps, manual edge-case verification). Architecture diagram present. Docker Compose brings the stack up. Dings: no `.dockerignore` (build context shipped `node_modules` + a stale committed `dist/`), `npm install` instead of `npm ci` in the Dockerfile (non-reproducible builds), stale `dist/` committed in the submission. |
| — | Tests | **Partially met** | What exists is above average for a take-home: the race test is a *real* race test, the fail-safe is tested against a hang (not just a refusal), and integration tests hit real Redis. But there was **not a single test of the HTTP layer** — routes, status codes, headers, validation were all untested (and indeed the API had an untested auth hole). The analytics pipeline — the billing path — had **zero** tests. Load test exists and asserts meaningful thresholds. |

## What was fixed (all verified by new tests that fail on the baseline)

1. **API-key auth on all `/v1` endpoints** (`src/api/auth.ts`, wired in `server.ts`): `x-api-key` must match `API_KEY`; `/health` and the dashboard's static files stay open; the dashboard sends the key and handles 401. k6 script updated accordingly.
2. **Unknown clientId → 404** (`src/api/routes.ts`): no more free default buckets for invented IDs.
3. **Self rate limiting** (`src/api/auth.ts`): per-source-IP fixed-window budget (default 600/min, `SELF_IP_LIMIT_PER_MINUTE`) on the service's own API, memory-bounded.
4. **Bounded stream** (`src/logging/eventStream.ts`): `XADD MAXLEN ~ 100000` (`ANALYTICS_STREAM_MAXLEN`) — a dead worker now means bounded event loss, never Redis OOM.
5. **Stranded-event recovery** (`src/logging/analyticsWorker.ts`): periodic `XAUTOCLAIM` sweep (60 s min-idle, 30 s cadence) adopts entries from dead consumers' pending lists.
6. **Idempotent billing inserts** (`analyticsWorker.ts` + `src/db/migrations/002_stream_id.sql`): every row carries the unique stream entry id; `ON CONFLICT (stream_id) DO NOTHING` makes crash-then-redeliver a no-op instead of a duplicate row.
7. **Honest fail-open contract** (`FailSafeRateLimiter.ts`): degraded responses now report the client's real configured limit from the in-memory registry (never `-1` sentinels), plus a sane `resetAtMs`.
8. **Graceful shutdown** (`server.ts`): SIGTERM/SIGINT drains Fastify, then closes Redis and the Postgres pool — now consistent with the worker.
9. **Runtime-editable limits** (`clientRegistry.ts`, compose bind-mount, `POST /v1/admin/clients/reload`): edit `clients.json`, hit reload, no rebuild. File is zod-validated; a broken edit keeps the previous registry (atomic swap).
10. **Dashboard**: request-count-weighted average latency, honest label ("limiter overhead"), granularity (day/hour) and outcome (all/allowed/denied) filters backed by new API query params, API-key field, 401 handling.
11. **Docker hygiene**: `.dockerignore` added; `npm ci` in both Dockerfile stages.
12. **Logging consistency**: remaining `console.*` call sites (event stream, worker, migrate) moved to the winston logger.

**New tests** (all red on the baseline, green after): `tests/unit/routes.test.ts` (11 HTTP-layer cases: 401/404/400/200+headers/429/degraded header/usage filters/reload/self-limit), `tests/unit/eventStream.test.ts` (MAXLEN + swallowed failure), `tests/unit/clientRegistry.test.ts` (validation, reload, reload-failure safety), two new `FailSafeRateLimiter` contract tests, and `tests/integration/analyticsWorker.integration.test.ts` (duplicate-batch idempotency and dead-consumer XAUTOCLAIM recovery against real Redis + Postgres).

**Post-fix verification**: `npm run test:unit` (27 tests), `npm run test:integration` (7 tests, real Redis + Postgres), `npm run typecheck`, and `docker compose up --build` all pass.

## Accepted / noted, not fixed

- `X-RateLimit-Reset` stays epoch-ms (documented in the type; changing it now would break the submitted contract for no functional gain).
- HALF_OPEN thundering herd on breaker recovery — real but low-impact at this scale; flagged for a follow-up.
- Timeout-path token over-consumption (request admitted via fail-open while the slow EVALSHA still lands) — bounded, degradation-only, defensible.
- Fastify's built-in pino request logging coexists with winston app logging — two log formats in one process is untidy but harmless.
- Chart.js from CDN — fine for a take-home; would vendor it for production.
- The `clients.json` bind-mount relies on in-place edits (an editor that replaces the file inode can detach a single-file bind mount) — acceptable for the exercise; production would use a DB-backed registry (the code's own comment already proposes exactly that).

## Overall verdict (as originally submitted, before fixes)

**Borderline fail — would not advance without a strong debrief.** The core rate-limiting engineering is genuinely good: the Lua-scripted token bucket, the hash-tagged keys, the circuit breaker with a hard timeout (covering the hang case, not just the refusal case), and a race-condition test that actually races are all above the bar, and the written explanations show the candidate understands *why* each piece exists. But the submission ships a **completely unauthenticated API in front of paid third-party quotas** — an instant production incident and by itself disqualifying for a service whose entire purpose is protecting metered budgets — and the analytics pipeline, described as billing-grade, can strand events forever, double-bill on crash, and grow Redis without bound until the limiter itself falls over. Those are not polish gaps; they are "does the candidate think about failure and abuse" gaps, and two of them are silent-data-loss/outage class. If the in-person round demonstrated the candidate understands these holes, the strong core would justify advancing; as a standalone artifact, it does not clear the bar.
